# Internationalization as Day-One Architecture

Retrofitting i18n means auditing every component for bare strings, untangling copy from concatenation and hand-rolled plurals, and re-testing every screen — typically a multi-week big-bang migration with a long tail of missed strings. Starting with i18n costs almost nothing: one macro habit, two npm scripts, and a lint rule. This file distills how to build a multilingual UI from the first commit, with an LLM-backed machine-translation pipeline so added languages cost minutes, not translator contracts.

## Principles

1. **Every user-facing string goes through a message macro from day one.** Not "we'll wrap them later" — later never comes cheaply. The macro (e.g. Lingui's `<Trans>` / `` t`` `` in React, as one instantiation; gettext and Fluent are the same idea) marks the string as copy, gives it an extractable identity, and makes it swappable at runtime.
2. **English (or one language) is the single source of truth.** Source-language copy lives *in the code*, readable in place; every other catalog is generated from it. There is no second place where English copy is edited.
3. **Extraction and translation are build steps, not projects.** `extract` collects macros into catalogs; `translate` fills the gaps; a `check` command fails the build if any locale has untranslated entries. The whole loop runs in one terminal session at the end of a feature branch.
4. **Catalogs are committed artifacts.** Translated catalogs live in git next to the code that uses them — reviewable in the diff, versioned with the feature, no runtime translation service, no external state.
5. **Hardcoded-string prevention is a lint rule, not a review habit.** A rule that flags raw UI strings (e.g. `lingui/no-unlocalized-strings`) is the mechanism that keeps principle 1 true after the tenth contributor. Humans reviewing for missed strings miss strings.
6. **Not everything translates.** Domain/technical terms your audience reads in English stay English, governed by a committed glossary — a deliberate editorial decision, not an omission.
7. **Language is a per-user preference, stored on the account.** Not a build variant, not a browser-only guess — the user chooses, the choice persists across devices, and both backend and frontend validate against one supported-locale list.

## Patterns

### 1. Message macros for every string shape

**Problem:** UI copy hides in many shapes — JSX text, `placeholder`/`title`/`aria-label` props, label maps in plain modules, pluralized counts built with `n > 1 ? 's' : ''`. Wrapping only the obvious JSX text leaves half the UI untranslatable, and string concatenation produces sentences that cannot be reordered for other grammars.

**Solution:** Adopt one macro per shape and document all four in the guide:

```tsx
<h1><Trans>Users</Trans></h1>                       // JSX text

const { t } = useLingui()                            // prop strings — via the hook,
<Input placeholder={t`Search…`} aria-label={t`Search`} />  // so locale changes re-render

<Plural value={n} one="# item" other="# items" />    // plurals — never hand-rolled

const STATUS = { failed: msg`Failed`, ok: msg`Passed` }    // label maps in plain .ts files:
<span>{i18n._(STATUS[status])}</span>                      // deferred msg, resolved at render
```

**Rationale:** Each shape has a failure mode: prop strings resolved outside the reactive hook don't update on locale switch; label maps evaluated at module load bake in one language forever (hence deferred descriptors resolved at render); hand-rolled plurals are simply wrong in most languages (Slavic languages have three-plus plural forms). Full messages with placeholders — never concatenated fragments — let translators reorder words.

### 2. The three-command developer loop

**Problem:** If translating requires a separate workflow, ticket, or person, translations lag the UI permanently and every release ships mixed-language screens.

**Solution:** Wire the loop as npm scripts and document it as the standard end-of-change routine:

```bash
# 1. write the macro in the component
npm run i18n:extract     # collect into source catalog; add EMPTY entries to every other locale
npm run i18n:translate   # fill only the empty entries via an LLM
# 2. commit code + updated catalogs together
npm run i18n:check       # CI/release gate: exit 1 if any locale has untranslated entries
```

The translate step must be **idempotent** — it fills only empty entries, so re-runs after extraction touch just the new strings — and must serialize with the same formatter the extractor uses, so output is byte-identical and diffs show only real changes, no churn.

**Rationale:** Making translation a two-command tail on every feature branch means catalogs are never stale by more than one PR, and the `check` gate makes "shipped untranslated" a build failure instead of a bug report from a user in another locale.

### 3. LLM-backed machine translation with audience context and a glossary

**Problem:** Human translation pipelines are slow and expensive for a fast-moving product; naive machine translation mangles domain vocabulary and register.

**Solution:** A small script fills empty catalog entries by calling an LLM (any OpenAI-compatible endpoint; key/base-URL/model via environment variables, nothing hardcoded). The prompt carries two committed pieces of context: the **audience and product description** (so register and terminology fit), and a **do-not-translate glossary** (JSON in the repo) listing product and technical terms that stay in the source language. The glossary is edited as the product vocabulary grows.

**Rationale:** For UI copy — short, contextual, domain-heavy — a well-prompted LLM with a glossary produces reviewable-quality output at near-zero marginal cost, which is what makes "i18n from day one" economically viable for a small team. Committed catalogs mean a human can still hand-correct any entry; idempotency means corrections survive (only *empty* entries are refilled).

### 4. The unlocalized-string lint rule, with a tuned escape hatch

**Problem:** After the macros exist, nothing stops the next contributor from typing a bare string — and the codebase silently accretes untranslatable copy again.

**Solution:** Enable a lint rule that flags raw strings in UI code. Since the rule can't distinguish copy from machine tokens, invest in its configuration: ignore lists for CSS values, route paths, machine keys, enum values, class-name helpers, plus type-aware exemptions for string-literal unions. For genuine false positives (query keys, thrown developer errors, sample data), require a one-line disable *with a stated reason*. Run the rule at `warn` while migrating an existing codebase; flip to `error` once clean — and on a greenfield project start at `error`.

**Rationale:** The rule is the enforcement arm of the whole architecture. The tuning matters: an untuned rule cries wolf on every route path and gets disabled globally, which is worse than not having it. The reasoned escape hatch keeps exceptions auditable.

### 5. One supported-locale list, validated on both sides

**Problem:** The set of supported languages ends up encoded in several places (backend validation, frontend catalog config, display names) that drift, producing users saved with locales the UI can't load.

**Solution:** Declare the supported-locale list in each tier that needs it — backend user validation and frontend catalog config — and document explicitly that they must stay in sync (or generate one from the other). Adding a language is then a three-step recipe: add the code to both lists plus a display name, run extract + translate to create and fill the catalog, commit the new catalog. **No component changes required** — that property is the proof the architecture is right.

**Rationale:** "New language = config + generated catalog, zero code" is the payoff of the whole system, and it doubles as a regression test: if adding a locale ever requires touching components, a string escaped the macro system.

### 6. Per-user language preference with a sane boot order

**Problem:** Browser-language detection alone guesses wrong and can't be corrected; storing the choice only client-side loses it across devices; loading the preference from the server after first paint flashes the wrong language.

**Solution:** Store the language on the user account (a BCP-47 code, validated against the supported list) with a small self-service endpoint. On the client, the authoritative source is the current-user query; a sync component applies it when it resolves and mirrors it to local storage as a *cache* for instant first paint. Boot order: local cache → server profile → browser language → source-language default. Put the picker somewhere every user can reach (e.g. the account menu), not inside an admin-only settings area. Unauthenticated/kiosk surfaces pin to the source language so public screens read consistently.

**Rationale:** Server-stored preference survives devices and reinstalls; the local-storage mirror kills the flash-of-wrong-language; the explicit fallback chain makes first-run behavior predictable instead of accidental.

### 7. Scope label translation apart from locale formatting

**Problem:** Conflating message translation with number/date/currency localization balloons the day-one scope and stalls both.

**Solution:** Treat them as separate concerns with separate decisions. Message catalogs are the day-one architecture; `Intl`-based number/date formatting can be a deliberate later step (a product may even *choose* a fixed date format for consistency). Similarly, decide explicitly which surfaces are in scope — the app UI, yes; a docs site may stay source-language-only.

**Rationale:** The expensive-to-retrofit part is the strings, because they are scattered through every component. Formatting is centralized in a couple of utility modules and is cheap to change later. Spend the day-one budget where retrofit cost is highest.

## Pitfalls

- **"We'll add i18n when we need it."** By then there are thousands of bare strings, concatenations, and hand-rolled plurals across every component — a big-bang migration with a long bug tail. The day-one cost is one habit and two scripts.
- **String concatenation and fragment translation.** `t('Hello') + ', ' + name + '!'` is untranslatable — word order differs by language. Always full messages with placeholders.
- **Plurals via ternary suffixes.** `count > 1 ? 's' : ''` is wrong in most of the world's languages. Use the plural macro from the first counted noun.
- **Label maps evaluated at module load.** A plain object of translated strings in a `.ts` file freezes one language at import time; use deferred message descriptors resolved at render.
- **Prop strings outside the reactive translation hook** — placeholders and aria-labels that don't update when the user switches language.
- **Translating the glossary.** Machine translation will happily localize your product's domain terms into unrecognizable words unless the glossary explicitly forbids it — and the glossary must be maintained as vocabulary grows.
- **Non-idempotent or reformatting translation tooling.** A translate step that rewrites whole catalogs churns every diff and clobbers hand corrections; fill empty entries only, serialize byte-identically.
- **An untuned lint rule.** Flagging every route path and CSS token trains the team to disable the rule. Tune ignores first, then enforce as `error`.
- **Uncommitted or generated-at-deploy catalogs.** Catalogs outside git can't be reviewed, bisected, or hand-corrected; translation becomes invisible infrastructure that silently regresses.

## Checklist for a new project

- [ ] Pick the i18n library and wire its build plugin (macro transform + catalog compilation) before the first screen ships.
- [ ] Declare the source language and initial locale list; create the catalog directory structure and commit it.
- [ ] Add the npm scripts: `i18n:extract`, `i18n:translate`, `i18n:check` — and make `check` a CI/release gate.
- [ ] Write the translation script: LLM-backed, env-configured, idempotent (empty entries only), byte-identical serialization, prompt includes audience/product context.
- [ ] Create the do-not-translate glossary file (even if nearly empty) and reference it from the translation prompt.
- [ ] Enable the unlocalized-strings lint rule at `error`, with tuned ignore lists and a documented reasoned-disable escape hatch.
- [ ] Document the four macro shapes (JSX text, prop strings via the hook, plurals, deferred label maps) in the contributor guide, with the "commit catalogs with your code" loop.
- [ ] Add the language field to the user model, validated against the supported list, with a self-service update endpoint.
- [ ] Implement the client boot order (cache → profile → browser → default) and place the language picker where every user can reach it.
- [ ] Write down the "adding a language" recipe and verify it: config + extract + translate + commit, zero component changes.
- [ ] Explicitly scope out (or in) number/date/currency formatting and non-app surfaces, so the decision is deliberate.
