# Internationalization (i18n)

Proxytrace's UI is multilingual. **English is the canonical source language**; other languages are
generated from it by an audience-aware translation tool. Each user picks their own UI language and
the choice is stored on their account.

Read this before adding any user-facing string or touching the translation system.

## Stack

- **[Lingui](https://lingui.dev) v6** — macros in the code are the source of truth; `lingui extract`
  collects them into `.po` catalogs. Configured in [`frontend/lingui.config.ts`](../frontend/lingui.config.ts).
- **Catalogs** live at `frontend/src/locales/{locale}/messages.po`, committed to git. `en` is the
  source (filled by extraction); other locales are filled by the translation tool.
- **Build** — the `@lingui/vite-plugin` compiles `.po` imports to runtime messages; the Lingui
  macros are transformed by `@rolldown/plugin-babel` + `linguiTransformerBabelPreset()` (Vite 8's
  oxc React plugin can't run inline Babel plugins). See [`frontend/vite.config.ts`](../frontend/vite.config.ts).
- **Runtime** — `frontend/src/i18n/index.ts` owns the shared `i18n` instance, the supported-locale
  list, and `dynamicActivate(locale)`. `main.tsx` wraps the app in `<I18nProvider>`.

The supported locales are declared in **two places that must stay in sync**:
`Proxytrace.Domain.User.SupportedLanguages` (backend validation) and `lingui.config.ts` `locales`
(frontend). Start set: `["en", "de"]`.

## Writing translatable UI strings

Use the Lingui macros — never a bare string in the UI.

```tsx
import { Trans, useLingui } from '@lingui/react/macro'  // components & hooks
import { msg } from '@lingui/core/macro'                // for label maps / outside components

// JSX text
<h1><Trans>Users</Trans></h1>

// Strings in expressions / props (placeholder, title, aria-label): use the `t` from useLingui so
// the component re-renders when the locale changes.
function Search() {
  const { t } = useLingui()
  return <Input placeholder={t`Search traces…`} aria-label={t`Search`} />
}

// Pluralization — never hand-roll `n > 1 ? 's' : ''`
import { Plural } from '@lingui/react/macro'
<Plural value={count} one="# proposal ready" other="# proposals ready" />

// Label maps in .ts/.tsx files: wrap with `msg`, resolve at render with i18n._()/t
const STATUS = { failed: msg`Failed`, passed: msg`Passed` }
const { i18n } = useLingui()
<span>{i18n._(STATUS[status])}</span>
```

**Do not translate** product/technical terms that read better in English for our AI-engineer
audience (Tool, User, Assistant, System, Trace, Token, Prompt, Agent, Proxy, Run, Suite, Evaluator,
Theory, Proposal, TTFT, …). The glossary is the source of truth — see below.

The ESLint rule `lingui/no-unlocalized-strings` (in `frontend/eslint.config.js`, scoped to `.tsx`)
flags raw UI strings and is an **`error`** — the big-bang migration is complete. The rule cannot tell
copy from non-copy tokens, so the config carries tuned `ignore` / `ignoreNames` / `ignoreFunctions`
lists (CSS values, route paths, snake_case/dotted machine keys, enum props, SVG geometry, class
helpers) plus `useTsTypes` for string-literal-union-typed values. For a genuine non-UI string the
rule still misfires on (enum tokens passed as values, query keys, thrown developer errors, Prism
token types, sample data, brand/confirmation strings), use the escape hatch:
`// eslint-disable-next-line lingui/no-unlocalized-strings -- <reason>`. Class strings go through
`cn()` (whitelisted) rather than a disable.

## The developer loop (adding/changing labels)

```bash
cd frontend
# 1. write the macro in your component (above)
npm run i18n:extract     # populates en/messages.po and adds empty entries to other locales
npm run i18n:translate    # fills the empty entries via the LLM (needs an API key, see below)
# 2. commit the updated catalogs alongside your code
```

`npm run i18n:check` fails (exit 1) if any locale still has untranslated entries — use it as a
release gate.

## The translation tool

[`frontend/scripts/i18n/translate.mjs`](../frontend/scripts/i18n/translate.mjs) fills empty
`msgstr` entries in every non-source catalog by calling an OpenAI-compatible model. It is
**idempotent** — only empty entries are translated, so re-runs after `i18n:extract` pick up just the
new strings. It parses/serializes with Lingui's own `@lingui/format-po`, so output is byte-identical
to `lingui extract` (no churn).

The model is given:
- the **audience** (AI engineers) and product context, and
- the **do-not-translate glossary** — [`frontend/scripts/i18n/glossary.json`](../frontend/scripts/i18n/glossary.json).
  Edit this file as the product's vocabulary grows.

Configure via environment (no keys are hardcoded; any OpenAI-compatible endpoint works, including
Proxytrace's own proxy):

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `I18N_TRANSLATE_API_KEY` | yes | — | API key for the endpoint |
| `I18N_TRANSLATE_BASE_URL` | no | OpenAI | endpoint base URL |
| `I18N_TRANSLATE_MODEL` | no | `gpt-4o-mini` | model id |

## Per-user language preference

- **Backend** — `IUser.Language` (BCP-47 culture code, default `en`); validated against
  `SupportedLanguages`. Self-service endpoint `PATCH /api/users/me` (`UpdateMyLanguageRequest`),
  returned in `MeDto`. See [`domain-entities.md`](domain-entities.md) and [`validation.md`](validation.md).
- **Frontend** — `useMe()` (`/api/auth/me`) is the authoritative source of the user's language;
  `LocaleSync` applies it once `/me` resolves and mirrors it to a localStorage cache for instant
  first paint. Boot order: cache → `/me` → browser language → English. The language picker lives in
  the **account dropdown in `Shell`** (`LanguageMenuItems`), reachable by every user — the settings
  hub is admin-only and cannot host a personal preference.

## Adding a new language

1. Add the code to `SupportedLanguages.All` (backend) and `locales` in `lingui.config.ts`, and add a
   display name to `LOCALE_NAMES` in `frontend/src/i18n/index.ts`.
2. `npm run i18n:extract` creates the empty catalog; `npm run i18n:translate` fills it.
3. Commit the new `frontend/src/locales/<code>/messages.po`. No component changes required.

> Number/date/currency formatting (`frontend/src/lib/format.ts`) is intentionally **out of scope**
> here — it keeps the design-mandated `dd.MM.yyyy` format. Locale-aware `Intl` formatting is a
> separate, future concern from label translation. The VitePress manual stays English-only.
